# Medical Applied Frontend

Готовый frontend-клиент для backend-приложения **Medical Applied** на **Next.js + React + TypeScript**.

## Что реализовано

Frontend покрывает весь реально реализованный REST API из загруженного backend-проекта:

- авторизация через JWT (`POST /api/login`, `POST /api/logout`)
- регистрация пользователя (`POST /api/users`)
- просмотр списка пользователей
- просмотр, обновление и удаление пользователя
- просмотр каталога препаратов с фильтрами по стране, категории и названию
- просмотр детальной карточки препарата
- просмотр истории просмотренных препаратов пользователя
- очистка истории просмотров пользователя
- создание, получение, обновление и удаление профиля здоровья пользователя
- создание запроса на верификацию email
- получение статуса верификации
- подтверждение email по токену
- смена пароля пользователя
- просмотр SEO-контента (`GET /api/seo/content`)

Также добавлена UX-заглушка для модуля записи к врачу, потому что в backend-коде контроллер есть, но REST-методы для него пока не реализованы.

## Дизайн-концепция

Интерфейс сделан как **операционная панель медицинского сервиса**:

- тёмная профессиональная палитра
- светящиеся акцентные карточки
- крупные секции для CRUD-операций
- быстрый доступ к основным сущностям: препараты, пользователи, SEO, профиль
- отдельная страница верификации email

## Стек frontend

- Next.js App Router
- React
- TypeScript
- CSS без UI-фреймворка, чтобы проект было проще адаптировать под твой backend

## Важные особенности backend API, которые frontend учитывает

### 1. Почти весь API защищён JWT

В backend `SecurityConfig` разрешает без токена только:

- `POST /api/login`
- `POST /api/users`
- Swagger

Поэтому каталог препаратов, SEO, список пользователей, верификация email и другие действия требуют авторизации.

### 2. Верификация email в текущем backend неудобна для UX

Метод подтверждения email:

- `GET /api/users/{id}/verify/{token}`

но он **тоже защищён JWT**. Поэтому перейти по email-ссылке без предварительного входа в систему не получится. Во frontend я добавил страницу `/verify`, которая умеет:

- прочитать `userId` и `token` из query params
- предложить сначала залогиниться
- после входа выполнить подтверждение

### 3. У профиля здоровья нет endpoint-а на список профилей

Есть только:

- создать профиль
- получить профиль по `healthId`
- обновить профиль по `healthId`
- удалить профиль по `healthId`

Из-за этого frontend:

- запоминает созданный `healthId` в `localStorage`
- позволяет вручную вставить `healthId`, если пользователь открыл сессию с другого браузера

Это не баг frontend — это ограничение текущего дизайна backend API.

## Требования

Для запуска проекта нужен современный Node.js, а структура приложения построена на каталоге `app` и стандартных пакетах `next`, `react` и `react-dom`.

## Установка

### 1. Перейди в каталог frontend

```bash
cd medical-applied-frontend
```

### 2. Установи зависимости

```bash
npm install
```

### 3. Создай локальный env-файл

```bash
cp .env.example .env.local
```

По умолчанию backend ожидается на:

```env
NEXT_PUBLIC_API_BASE_URL=http://localhost:8090
```

Если backend запущен на другом хосте или порту — измени это значение.

### 4. Запусти dev-сервер

```bash
npm run dev
```

После запуска frontend будет доступен по адресу:

```text
http://localhost:3000
```

Используется стандартный workflow Next.js: dev для локальной разработки, build для production-сборки и start для запуска собранного приложения.

## Production-сборка

```bash
npm run build
npm run start
```

Это стандартные production-команды для сборки и запуска приложения.

## Как пользоваться

### Первый сценарий: регистрация и вход

1. Открой `/register`
2. Создай пользователя
3. Перейди на `/login`
4. Введи email и пароль
5. После входа откроется dashboard

### Работа с препаратами

1. Перейди на `/medicals`
2. Выбери страну, категорию и/или название
3. Нажми «Найти препараты»
4. Открой карточку препарата

> Важно: просмотр детали препарата на backend записывает препарат в историю просмотров пользователя, если запрос сделан с JWT.

### Работа с пользователями

1. Перейди на `/users`
2. Выбери пользователя
3. На странице пользователя доступны:
   - обновление профиля
   - удаление пользователя
   - история просмотров препаратов
   - операции с профилем здоровья
   - email verification
   - смена пароля

### Работа с профилем здоровья

1. На странице пользователя создай профиль здоровья
2. Система запомнит `healthId` локально
3. Для последующих `get/update/delete` этот `healthId` будет подставляться автоматически
4. При необходимости его можно вставить вручную

### Верификация email

Есть два сценария:

#### A. Через профиль пользователя

- нажать «Создать запрос на верификацию»
- проверить статус
- вставить токен вручную и выполнить подтверждение

#### B. Через страницу `/verify`

Открыть адрес вида:

```text
/verify?userId=<UUID>&token=<UUID>
```

Если пользователь не вошёл, страница предложит сначала авторизоваться.

## Что стоит улучшить в backend после подключения frontend

1. Добавить `GET /api/users/me`
2. Сделать публичным endpoint подтверждения email или переводить письмо на frontend route
3. Добавить endpoint списка / текущего профиля здоровья пользователя
4. Добавить реальные endpoints для записи к врачу
5. Разделить роли USER / ADMIN на уровне UI и backend authorization rules

## Быстрый запуск backend + frontend

Backend у тебя по умолчанию слушает порт `8090`, а frontend — `3000`. Этого достаточно для локальной разработки, если CORS разрешён с frontend-хоста.

Если CORS ещё не настроен, добавь его в Spring Boot для `http://localhost:3000`.
